# AUDITORIA PRODUCTION-READY — 2026-04-06

> Reavaliação completa do Django Shopman pós-CORRECTIONS-PLAN.
> Foco: fluxos E2E do cliente, alinhamento storefront↔backend, POS UX, gaps para produção.

---

## 1. VEREDICTO GERAL

O backend é **maduro e feature-rich**. O storefront **subutiliza ~30%** das capacidades do backend. O POS é um **MVP funcional** mas longe de production-ready. Os testes cobrem happy paths mas têm **gaps críticos** em webhooks, concorrência e error paths.

**Estado por eixo:**

| Eixo | Estado | Nota |
|------|--------|------|
| Storefront E2E (happy path) | ✅ Sólido | Menu→Cart→Checkout→PIX→Tracking funciona |
| Storefront edge cases | ⚠️ Parcial | Faltam: delivery fee, zona de entrega, alternativas |
| POS | ⚠️ MVP | Funcional mas primitivo para operação real |
| KDS | ✅ Bom | Dispatch por estação, timers, check/done |
| Gestor de Pedidos | ⚠️ Funcional | Confirm/advance/reject OK, falta refinamento |
| Backend capabilities | ✅ Robusto | 70% usado, 30% pronto mas não exposto |
| Testes | ⚠️ Parcial | ~1.900 testes, gaps em webhooks, concurrency, E2E real |
| Pagamento cartão | ❌ Não exposto | Backend Stripe pronto, falta UI no checkout |
| Delivery fee | ❌ Inexistente | Nenhuma menção no código |
| Loyalty UI (resgate) | ❌ Não exposto | Backend completo, nenhum UI de resgate |

## 2. FLUXOS DO CLIENTE — O QUE É FIRST-CLASS E O QUE NÃO É

### 2.1 ✅ FIRST-CLASS: Pedido Web com PIX

```
Menu → Busca/Filtro → Detalhe → Add to Cart → Cart (qty/coupon) → Checkout → PIX QR → Tracking
```

- Disponibilidade com badges (D-1, preparando, esgotado, planejado)
- Preço com modifiers (D-1 50%, promoções auto, cupons, "melhor desconto ganha")
- Checkout com phone E.164, endereço estruturado + CEP lookup, cutoff, pré-encomenda
- Commit idempotente (IdempotencyKey 24h TTL)
- Stock hold com TTL
- Tracking com timeline de eventos, cancelamento self-service, reorder
- Confirmação otimista com countdown
- Polling via HTMX (status, pagamento)

**Avaliação:** Fluxo completo e robusto. Passageiro de primeira classe.

### 2.2 ⚠️ PARCIAL: Pedido Web com Cartão

O backend tem `StripeBackend` + `StripeWebhookView` + `ChannelConfig.Payment` aceita `method: ["pix", "card"]`. O checkout lê `payment_methods` do canal e mostra seletor se >1 método.

**Gap:** A payment page não tem branch para Stripe Elements. Se o cliente selecionar "cartão", não há template de input de cartão. O routing pós-checkout (`PIX → QR page` vs `Card → Stripe confirmation`) não existe.

### 2.3 ⚠️ PARCIAL: Delivery vs Pickup

- **Pickup:** ✅ funciona, tracking mostra endereço da loja + horário
- **Delivery:** endereço estruturado + CEP lookup (ViaCEP) + endereços salvos + fulfillment tracking

**Gaps fundamentais:**
- **Nenhuma taxa de entrega** — `delivery_fee` não existe em nenhum lugar do código. Grep retorna zero.
- **Nenhuma validação de zona de entrega** — aceita qualquer CEP do Brasil
- **Nenhum cálculo de distância/raio** — não há geofencing
- Para produção com delivery, isso é **bloqueante**.

### 2.4 ✅ BOM: Auth do Cliente

- OTP por WhatsApp/SMS com fallback automático
- Device trust: cookie HMAC 30 dias, skip OTP on repeat
- Access link (WhatsApp): token → auto-login
- Rate limiting: 5/m request, 10/m verify, 3/m login POST
- Multi-step UI (phone → code → name para novos)

### 2.5 ✅ BOM: Conta do Cliente

- Dashboard com pedidos recentes, loyalty (se instalado)
- Endereços salvos (CRUD + set default + labels)
- Perfil editável + preferências alimentares + prefs de notificação
- Export GDPR + delete account
- Device management (list + revoke)

### 2.6 ❌ NÃO TRATADO: Devolução/Troca

- `returns.py` handler existe no backend
- `EVENT_LABELS` tem "return_initiated" e "refund_processed"
- **Nenhuma UI** para iniciar devolução — nem storefront, nem gestor

### 2.7 ❌ NÃO TRATADO: Pagamento no Balcão (counter)

- `payment_method: "counter"` é o default do POS
- Mas **não existe UI para operador marcar "pago"** no gestor de pedidos
- Order com payment counter não tem nenhum trigger de `payment.captured`
- O pedido avança por status (confirm → processing → ready) sem validar pagamento

## 4. POS — ESTADO ATUAL E REDESIGN

### 4.1 O que o POS faz hoje

- Grid de produtos filtráveis por coleção (F1-F4)
- Cart no sidebar com ±qty via Alpine.js
- Customer lookup por telefone (HTMX POST → retorna nome/ref)
- Seleção de pagamento (dinheiro/PIX/cartão — só label, sem captura real)
- Close sale → Session → ModifyService → CommitService → Order
- Feedback de sucesso com ref + total (4s auto-dismiss)

### 4.2 Gaps para produção

**Bloqueantes:**

| Item | Descrição |
|------|-----------|
| **Cancelamento** | Sem botão de cancelar venda em andamento ou última venda |
| **Desconto manual** | Operador não pode aplicar desconto ad-hoc |
| **Gestão de caixa** | Sem abertura, sangria, fechamento, troco |
| **Observação por item** | Sem campo de personalização (ex: "sem açúcar") |
| **Histórico do turno** | Operador não vê o que já vendeu |

**UX Issues:**

| Item | Problema |
|------|----------|
| Grid estático | Produtos carregados 1x no GET, sem refresh se preço/stock muda |
| Sem atalhos completos | F1-F4 para coleções existem, falta F8=fechar, Esc=limpar, etc |
| Erro genérico | `Erro: {e}` — não distingue stock, payment, validation |
| Sem badge D-1 | Menu não mostra D-1 como o storefront |
| Sem employee discount toggle | Modifier existe mas operador não ativa |
| Sem estado offline | Server down = POS morto |
| Sem resumo de turno | Não sabe quantas vendas fez nem total |

### 4.3 Redesign proposto

**Princípio:** Velocidade > estética. Cada venda simples em <15 segundos.

```
┌──────────────────────────────────────────────────────────────┐
│ [🔍 Busca]  [Todas] [Pães] [Doces] [Bebidas]  [⚙️]        │
├──────────────────────────────────┬───────────────────────────┤
│                                  │  VENDA #                  │
│   ┌────┐ ┌────┐ ┌────┐ ┌────┐  │  ─────────────────────── │
│   │Prod│ │Prod│ │Prod│ │Prod│  │  Pão Francês    2x  3,00 │
│   │ 1  │ │ 2  │ │D-1 │ │ 4  │  │  Café Exp.     1x  5,00 │
│   │3,00│ │5,00│ │2,50│ │4,00│  │  [Obs: sem açúcar]       │
│   └────┘ └────┘ └────┘ └────┘  │                           │
│   ┌────┐ ┌────┐ ┌────┐ ┌────┐  │  ─────────────────────── │
│   │    │ │    │ │    │ │    │  │  Subtotal:         11,00  │
│   │    │ │    │ │    │ │    │  │  Desconto Staff:   -2,20  │
│   │    │ │    │ │    │ │    │  │  TOTAL:             8,80  │
│   └────┘ └────┘ └────┘ └────┘  │                           │
│                                  │  📱 (43) 99999-1234      │
│                                  │  → João Silva (STAFF) ★  │
│                                  │                           │
│                                  │  [💵 Din] [◻ PIX] [💳]   │
│                                  │  [% Desc]  [📝 Obs]      │
│                                  │                           │
│                                  │  ╔═══════════════════════╗│
│                                  │  ║   FECHAR VENDA (F8)  ║│
│                                  │  ╚═══════════════════════╝│
├──────────────────────────────────┴───────────────────────────┤
```

**Atalhos:** F1-F4=coleções, F5=limpar, F6=desconto, F7=obs, F8=fechar, Esc=cancelar, `/`=busca

**Melhorias-chave:**
1. Badge D-1 nos tiles (já existe no storefront, replicar)
2. Desconto manual: modal com % ou valor + motivo obrigatório
3. Obs por item: click no item → campo inline
4. Footer com resumo de turno (vendas + total)
5. Employee discount auto-highlight quando cliente é staff
6. Histórico: drawer lateral com últimas vendas

## 5. TESTES — GAPS PARA PRODUÇÃO

### 5.1 Bem coberto (~1.900 testes)
- Order lifecycle completo (session → commit → transitions)
- Payment state machine com row-level locking (concurrent capture/refund)
- Stock holds (create, fulfill, release, bundle expansion)
- Auth (OTP, device trust, access links, rate limiting)
- Modifiers pipeline (D-1, discount, employee, happy hour)
- Security headers
- Checkout web views (happy path)

### 5.2 Gaps críticos

| Gap | Risco | Prioridade |
|-----|-------|------------|
| **Webhook payment (Stripe/EFI)** | PIX pago mas order não avança | 🔴 Crítica |
| **Falha de serviço externo** | Gateway down = order travado, sem fallback | 🔴 Crítica |
| **Checkout E2E real** | Playwright testa page loads, não o fluxo completo | 🔴 Crítica |
| **Concurrent checkout** | 2 users comprando último item = oversell? | 🟡 Alta |
| **Webhook idempotency** | Gateway envia callback duplicado | 🟡 Alta |
| **Cart price drift** | Item repriced entre add-to-cart e checkout | 🟡 Alta |
| **Error paths nas views** | Cart inválido, session expirada, preço mudou | 🟡 Média |
| **POS/KDS/Pedidos views** | Zero testes para fluxos de operador | 🟡 Média |
| **Session timeout mid-checkout** | 30min TTL, comportamento desconhecido | 🟡 Média |
| **N+1 query assertions** | Sem baseline de performance | 🟠 Baixa |

## 7. RESUMO EXECUTIVO

**O Django Shopman está ~75% production-ready para cenário pickup-only com PIX.** O fluxo principal (web → PIX → tracking) é sólido, idempotente e bem testado.

**Para delivery real**, faltam 3 fundamentos: taxa de entrega, zona de entrega, pagamento com cartão. Sem esses, delivery é "aceita qualquer endereço, grátis, só PIX".

**Para operação de loja com POS**, o MVP serve para demo mas falta o básico diário: cancelamento, desconto manual, gestão de caixa, histórico.

**O backend está pronto para mais do que a UI oferece.** Loyalty resgate, RFM targeting, alternativas, happy hour visual, checkout defaults — são quick wins que dependem só de UI.

**O ponto cego mais perigoso para produção:** webhooks de pagamento não testados. Um PIX pago que não transiciona o pedido é o bug mais provável em produção real.

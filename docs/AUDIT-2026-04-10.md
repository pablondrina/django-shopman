### B3 — Doorman hard dependency em Guestman

**Arquivo:** `packages/doorman/pyproject.toml`
**Descrição:**
```toml
dependencies = [
    "Django>=5.2,<6.0",
    "shopman-guestman>=0.1.0",   # ← problema
]
```
Auth deveria ser o package mais fácil de usar standalone. Qualquer projeto que use
Doorman (ex: B2B sem CRM) é forçado a instalar todo o Guestman.

O `CUSTOMER_RESOLVER_CLASS` padrão aponta para `shopman.guestman.adapters.auth.CustomerResolver`,
o que concretiza a dependência em runtime.

**Ação:**
- Criar `shopman.doorman.resolvers.NoopCustomerResolver` (levanta `NotImplementedError` com mensagem clara)
- Mudar default de `CUSTOMER_RESOLVER_CLASS` para o noop
- Remover `shopman-guestman` do `requires` (manter apenas como optional/extra)
**Severidade:** Média — viola o princípio de standalone dos core apps.

### B7 — Guestman __init__.py exporta 20+ símbolos

**Arquivo:** `packages/guestman/shopman/guestman/__init__.py`
**Descrição:** `__all__` declara 20 símbolos (3 de Gates + 17 de contribs).
Para comparação, Offerman exporta 4 e Craftsman exporta 5.

Os contribs são APIs secundárias — forçar importação direta (`from shopman.guestman.contrib.loyalty import LoyaltyService`)
é mais explícito e consistente com os outros packages.

**Ação:** Restringir `__all__` para `["Gates", "GateError", "GateResult"]`.
O `__getattr__` com lazy imports pode permanecer como conveniência, mas fora do `__all__`.
**Severidade:** Baixa — questão de API surface e consistência.

### B11 — FiscalBackend/AccountingBackend não documentados como extension points

**Arquivos:**
- `packages/orderman/shopman/orderman/protocols.py` — `FiscalBackend`, `AccountingBackend`
- `framework/shopman/fiscal.py` — `FiscalBackend` base class, `FiscalPool`
- `framework/shopman/handlers/fiscal.py`, `framework/shopman/handlers/accounting.py`

**Descrição:** Os Protocols existem, os handlers existem, os mocks existem — mas não há
indicação clara de que estas são extensões que o integrador deve implementar para produção,
ao contrário de outros adapters (payment, notification) que têm exemplos explícitos.

**Ação:** Adicionar docstring explícita nos Protocols indicando:
- Que são extension points intencionais (não bugs)
- Quais implementações de referência existem (FocusNFCeBackend, ContaAzulBackend)
- Que o default (ausência de backend) é silencioso/noop — o sistema funciona sem eles
**Severidade:** Baixa — qualidade de documentação para integradores.
